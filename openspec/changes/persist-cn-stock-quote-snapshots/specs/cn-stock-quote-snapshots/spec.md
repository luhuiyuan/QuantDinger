## ADDED Requirements

### Requirement: 持久化每只 A 股的最新有效行情
系统必须（SHALL）为每只启用的沪深 A 股最多保存一条最新展示行情，至少包含规范标识、代码、交易所、最新价、昨收、涨跌额、涨跌幅、开高低、成交量、成交额、上游行情时间、来源和写入批次。系统不得把 5 分钟刷新结果追加为无限增长的分钟历史表。

#### Scenario: 同一股票取得更新报价
- **WHEN** 后台任务取得同一规范标识的更新有效报价
- **THEN** 系统以批量 UPSERT 更新该股票最新快照且表中仍只有一条当前记录

#### Scenario: 新报价无效或获取失败
- **WHEN** 新报价缺少有效身份或核心价格字段，或该股票所在腾讯批次失败
- **THEN** 系统不使用空值或零值覆盖数据库中的上一次有效快照

### Requirement: 每五分钟主动刷新交易时段行情
系统必须（SHALL）由后台调度在 A 股交易日的交易时段每 5 分钟触发行情刷新，并在收盘后执行一次最终刷新。任务必须自行验证交易日和会话，不得依赖前端访问或刷新按钮启动。

#### Scenario: 交易时段无人打开页面
- **WHEN** 当前为 A 股交易日连续竞价时段且没有用户访问前端
- **THEN** 后台仍按 5 分钟计划刷新并持久化最新行情

#### Scenario: 午休或非交易日触发调度
- **WHEN** Beat 在午休、夜间或非交易日触发任务
- **THEN** 任务记录或返回跳过结果且不发起全市场行情请求

#### Scenario: 收盘最终刷新
- **WHEN** 当日收盘后的最终刷新窗口到达
- **THEN** 系统最多执行一次当日最终刷新并将有效快照标记为已收盘数据

### Requirement: 全市场主源失败时自动使用腾讯分批降级
系统必须（SHALL）优先尝试受时限保护的完整市场快照；主源失败、超时、为空或覆盖率不足时，必须从启用目录按稳定顺序分批调用腾讯报价，每批最多 100 只且默认串行执行。系统不得退化为逐股票网络请求。

#### Scenario: 完整市场快照可用
- **WHEN** 主源在时限内返回满足覆盖要求的规范化快照
- **THEN** 系统分批持久化该快照且不再执行腾讯全目录降级

#### Scenario: 主源超时后腾讯批次可用
- **WHEN** 完整市场快照超时但腾讯批量报价可访问
- **THEN** 系统串行处理全部目录批次、提交成功批次并记录来源为腾讯降级或混合来源

#### Scenario: 部分腾讯批次失败
- **WHEN** 部分腾讯批次成功而另一些批次失败或未返回个别股票
- **THEN** 成功批次保持提交，失败股票保留旧快照，刷新运行标记为部分成功并记录各类数量

### Requirement: 防止重复和重叠刷新
系统必须（SHALL）使用具有所有权和过期时间的分布式锁，并结合持久化运行状态保证同一时刻最多有一个有效的 A 股快照刷新。Worker 重启、重复 Beat 或手工触发不得产生无界并发任务。

#### Scenario: 上一轮尚未完成
- **WHEN** 下一次 5 分钟调度到达但已有刷新持有有效锁
- **THEN** 新任务安全跳过并记录重复触发，不再启动第二轮全市场请求

#### Scenario: Worker 异常退出
- **WHEN** 持锁 Worker 异常退出且无法主动释放锁
- **THEN** 锁在配置 TTL 后可恢复，后续任务能够重新执行且不会删除已有有效快照

### Requirement: 基于数据库快照排序筛选分页
系统必须（SHALL）允许 A 股目录按代码、名称、涨跌幅、成交量、成交额和行情时间执行白名单排序，并基于数据库最新快照完成涨跌状态筛选和服务端分页。动态排序必须使用稳定次排序及 `NULLS LAST`，响应总数必须对应完整数据库查询结果。

#### Scenario: 按成交额降序浏览
- **WHEN** 用户请求按成交额降序排列第一页
- **THEN** 系统从数据库返回成交额最高的可用快照，缺失报价位于可用结果之后且分页顺序稳定

#### Scenario: 筛选深市上涨股票
- **WHEN** 用户选择 `SZ`、上涨状态并按涨跌幅降序
- **THEN** 总数和分页结果基于数据库中全部匹配快照计算，而不是只基于当前页网络报价

#### Scenario: 请求未授权排序字段
- **WHEN** 客户端提交不在白名单中的排序字段或方向
- **THEN** 系统返回结构化参数错误或使用明确默认排序，不拼接未验证 SQL

### Requirement: 明确展示数据库行情的新鲜度和覆盖
系统必须（SHALL）基于每只股票的上游 `quote_time` 和 A 股交易会话判断 `fresh`、`stale`、`closed` 或 `unavailable`，并返回目录数、快照覆盖数、最新/最旧行情时间、数据来源和最近刷新状态。系统不得使用数据库写入时间或页面刷新时间冒充行情时间。

#### Scenario: 交易时段快照超过新鲜阈值
- **WHEN** 当前处于交易时段且某行情时间距当前超过配置的新鲜阈值
- **THEN** 该行情仍可展示但明确标记为陈旧并显示真实行情时间

#### Scenario: 当日最终刷新完成
- **WHEN** 当前已收盘且股票具有当日最终有效行情
- **THEN** 系统将其标记为 `closed`，不因夜间经过超过 10 分钟而错误标记为陈旧

#### Scenario: 数据库尚无任何快照
- **WHEN** migration 后首次后台刷新尚未成功
- **THEN** 列表可以使用现有逐页报价灾难降级，但必须返回 `partial=true` 且不得宣称全市场排序筛选完整

### Requirement: 展示快照不得改变权威回测边界
持久化的最新行情必须（SHALL）仅用于市场列表和详情报价展示，不得作为权威历史、技术指标日线、复权因子或回测覆盖依据。

#### Scenario: 用户从数据库行情进入回测
- **WHEN** 用户从 A 股列表或详情携带标的进入回测中心
- **THEN** 回测仍使用现有权威日线覆盖检查，不因存在最新行情快照而绕过门禁

### Requirement: 前端读取不得触发周期性行情刷新
A 股列表和详情前端必须（SHALL）在首次进入、查询条件变化或用户明确刷新时读取数据库快照，不得使用 `setInterval`、页面可见性事件或其他前端定时器周期性请求行情。数据库已有覆盖时，概览和详情 API 不得在用户请求链路等待完整市场上游刷新。

#### Scenario: 用户停留在列表页面
- **WHEN** 用户打开 A 股列表后保持页面可见且不进行任何操作
- **THEN** 前端不再每 30 秒或其他固定间隔请求行情，后台 5 分钟任务独立更新数据库

#### Scenario: 用户手动点击刷新
- **WHEN** 用户明确点击刷新按钮
- **THEN** 前端重新读取数据库最新快照，但不在该请求中启动全市场网络刷新

### Requirement: 后台同步生成市场概览缓存
每轮后台行情刷新完成后，系统必须（SHALL）基于持久化股票快照计算市场宽度、获取核心指数并把完整市场概览写入 Redis。市场概览 API 必须优先读取该缓存，不得为每个前端请求重复扫描全市场并请求指数。

#### Scenario: 后台行情刷新成功
- **WHEN** 一轮后台行情刷新完成并已提交有效股票快照
- **THEN** 同一任务生成包含指数、涨跌统计、成交额、覆盖数、来源和行情时间的 Redis 概览缓存

#### Scenario: Redis 概览缓存丢失
- **WHEN** 用户请求概览但 Redis 中不存在缓存且 PostgreSQL存在有效快照
- **THEN** 系统从 PostgreSQL重建并回填一次缓存，后续请求直接读取缓存
