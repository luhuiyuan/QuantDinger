## Context

现有 A 股页面以 `qd_market_symbols` 作为目录，以 Redis 中的全市场快照作为动态行情。页面可见时每 30 秒请求，后端在缓存过期后按需刷新；无人访问时不会主动更新。部署网络中的 Eastmoney/AkShare 全市场快照经常超过 12 秒超时，腾讯逐页报价虽能维持页面可用，却无法对约 5,000 只沪深 A 股执行可信的全市场排序、涨跌筛选和分页。

项目已有 PostgreSQL、Redis、Celery Worker/Beat、A 股交易日历、腾讯批量报价和低资源串行执行约束。本变更需要跨数据库 migration、后台调度、行情服务、列表 API 和 Vue 页面，同时不得把展示行情误认为权威日线或回测输入。

## Goals / Non-Goals

**Goals:**

- 为每只启用的沪深 A 股持久化最近一次有效展示行情，并可在 PostgreSQL 中完成排序、筛选和分页。
- 在交易日连续竞价时段每 5 分钟主动刷新，收盘后执行一次最终刷新，不依赖前端访问触发。
- 全市场主源失败时，以最多 100 只一批串行使用腾讯报价，允许批次部分成功并保留旧值。
- 提供任务互斥、幂等、运行记录、错误摘要和数据新鲜度，使部署与故障可观察。
- 保持低内存、低并发和现有 A 股日线/回测覆盖门禁不变。

**Non-Goals:**

- 不保存每 5 分钟一次的历史序列，不建设分钟行情仓库。
- 不把展示快照用于回测、复权、技术指标或成交模拟。
- 不承诺交易所级实时性；目标是约 5 分钟级展示行情。
- 不在本变更中引入新的付费行情源，也不调整 `easy_tdx` 开发期 wheel 基线。
- 不为北交所、港股或美股扩展相同能力。

## Decisions

### 1. 使用最新状态表而不是追加式时间序列表

新增 `qd_cn_stock_quote_snapshots`，以规范 `instrument` 为唯一键，保存代码、交易所、最新价、昨收、涨跌额、涨跌幅、开高低、成交量、成交额、上游行情时间、来源、本次刷新批次和更新时间。每个成功报价通过 PostgreSQL `INSERT ... ON CONFLICT DO UPDATE` 覆盖最新状态。

为 `change_percent`、`volume`、`amount`、`quote_time` 建立适合 `WHERE market/exchange` 与排序的索引。目录身份仍以 `qd_market_symbols` 为准，查询使用左连接，确保没有报价的启用证券仍可出现并标记不可用。

备选方案是每 5 分钟追加一行；未采用，因为首期只需要最新排序，追加约 5,000×48 条/日会快速消耗当前仅约 3 GiB 的磁盘余量，并引入分区、保留和压缩运维。

### 2. 每个批次独立提交，永不以失败值覆盖有效快照

刷新服务先规范化并校验报价；缺少有效最新价、昨收或行情身份的行不进入 UPSERT。腾讯降级按最多 100 只一批串行处理，每批在单独短事务中批量提交。后续批次失败时，已成功批次保持提交；失败股票保留数据库中上一次有效记录。

`quote_time` 表示上游行情时间，`updated_at` 表示数据库写入时间。前端及 API 新鲜度只能依据 `quote_time`、交易日历和任务状态判断，不得使用 `updated_at` 或页面刷新时间冒充行情时间。

备选方案是全市场单事务；未采用，因为一次网络或数据库错误会回滚大量已取得报价，并形成长事务和更高内存峰值。

### 3. Celery Beat 负责触发，任务内部再次验证交易时段

Celery Beat 每 5 分钟触发轻量任务。任务调用现有 A 股市场日历判断当前是否为交易日及交易时段；上午、下午连续竞价期间执行刷新，午休和非交易日快速跳过。收盘后设置一个最终刷新窗口，使用当日完成标记保证只执行一次。

Beat 调度不是唯一安全边界：任务取得 Redis 分布式锁并检查数据库中的运行批次，避免多个 Beat、Worker 重启或手工触发造成重复刷新。锁具有大于预期执行时间的 TTL，并通过所有权 token 安全释放；超时任务由运行记录暴露，不自动无限并发补跑。

### 4. 完整快照优先，腾讯目录分批作为自动降级

每轮先调用现有标准化全市场快照 provider。若在配置时限内返回有效覆盖，则直接分批 UPSERT。若失败、超时、返回空集或覆盖率低于配置阈值，则从 `qd_market_symbols` 读取全部启用沪深 A 股，按稳定代码顺序每批最多 100 只调用 `fetch_quote_map`。

腾讯批次串行执行并复用现有限流器、重试和请求超时。任务记录实际来源为 `full_snapshot`、`tencent_batch` 或 `mixed`，以及计划、成功、失败和未返回数量。不得为每只证券发起单独网络请求。

### 5. 数据库查询成为列表排序筛选的稳定基线

`GET /api/market/cn/stocks` 从目录左连接最新快照，允许白名单排序字段：代码、名称、涨跌幅、成交量、成交额和行情时间，并允许明确的升降序。所有动态排序使用稳定的代码次排序及 `NULLS LAST`，确保翻页不重复或漂移。

涨跌状态筛选基于数据库 `change_percent`，因此其 `total` 表示当前数据库快照中满足条件的完整目录结果，而不是当前页腾讯报价子集。响应返回快照覆盖数、目录数、最新/最旧行情时间、任务状态和整体新鲜度。数据库完全无快照时，保留现有逐页降级作为灾难恢复，但明确 `partial=true` 且不宣称全市场排序完整。

### 6. Redis 只承担加速与协调，PostgreSQL 是最近行情基线

列表查询结果可以短 TTL 缓存在 Redis，缓存键包含筛选、排序、分页及数据库快照版本。每个成功写入批次更新版本并使旧查询缓存自然失效。Redis 丢失不影响数据库最新行情；PostgreSQL 写入失败则不得只把新行情留在 Redis 并宣称持久化成功。

### 7. 新鲜度按交易会话解释

交易时段内，最新有效 `quote_time` 距当前不超过 10 分钟标记为 `fresh`，超过 10 分钟标记为 `stale`；缺少报价标记为 `unavailable`。收盘后，当日最终刷新完成的数据标记为 `closed`，不因夜间时间流逝被误报为陈旧。阈值可配置，但前端必须显示实际行情时间和来源。

个股详情仍可在数据库快照基础上请求腾讯近实时报价；该覆盖仅用于详情展示，不立即改变全市场数据库排序基线，也不获得回测资格。

列表页和详情页不得使用 `setInterval` 或页面可见性事件定时拉取行情。默认读取只发生在首次进入、筛选/排序/分页变化和用户明确点击刷新时；网络行情刷新完全由后台任务负责。概览和详情 API 同样优先读取 PostgreSQL，数据库有覆盖时不得在用户请求链路等待完整市场上游。

后台行情批次完成后必须基于同一 PostgreSQL 快照计算市场宽度、批量获取三个核心指数，并把完整概览写入 Redis。`/cn/overview` 正常只读取该缓存；Redis 缓存丢失时允许从 PostgreSQL重建并回填一次，但不得在每个用户请求中重复扫描全市场和请求指数。

### 8. 独立运行记录支持验收和运维

新增 `qd_cn_stock_quote_refresh_runs`，记录任务类型、计划数、成功数、失败数、未返回数、来源、状态、开始/结束时间、最后错误和触发方式。错误摘要必须限长且不得包含凭据。任务状态可由受登录保护的市场 API 返回摘要；若增加手工触发，只允许管理员且复用同一锁和调度判断。

## Risks / Trade-offs

- [约 5,000 只股票的腾讯降级需要约 50 次批量请求] → 每批最多 100、严格串行、复用限流器并记录耗时；不缩短到高频秒级调度。
- [5 分钟任务可能尚未结束下一轮又到达] → Redis 所有权锁、运行批次检查和跳过计数共同防重。
- [不同批次行情时间不完全一致] → 保存每只股票自身 `quote_time`，同时返回全局最旧/最新时间和覆盖率，不伪装成原子全市场切片。
- [数据库排序索引增加写放大] → 只保留最新一行并使用必要的有限索引；实现后用执行计划验证常用查询。
- [数据库或磁盘空间不足] → migration 前检查磁盘；任务写入失败保留旧数据并报告，不自动清理 Docker、缓存或项目数据。
- [停牌股票长期没有新成交] → 以有效报价和交易日期判断，不用零价覆盖；目录行仍可展示最近行情及状态。

## Migration Plan

1. 新增幂等 migration，创建快照表、运行记录表和索引；现有表不做破坏性修改。
2. 实现仓储、刷新服务和聚焦测试，通过手工一次性刷新验证覆盖率、批量 UPSERT 和查询执行计划。
3. 调整列表 API 和前端排序/新鲜度展示，同时保留数据库空表时的现有逐页降级。
4. 部署 backend、Celery Worker 和 Celery Beat，确认只出现一个有效刷新任务；再部署 frontend。
5. 观察至少一个交易时段的 5 分钟任务、午休跳过、收盘最终刷新和腾讯降级记录。
6. 回滚时先禁用 Beat 调度，再回滚 API/前端读取路径；新增表可保留，不影响原有目录、日线和回测，删除表需另行明确批准。

## Open Questions

无阻断问题。首期排序白名单固定为代码、名称、涨跌幅、成交量、成交额和行情时间；以后新增市值、换手率等字段需另行扩展数据源和规格。
