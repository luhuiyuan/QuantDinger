## Context

当前后端已经有 `/api/market` 下的标的搜索、报价和自选股接口，`qd_market_symbols` 保存标的主数据，腾讯源可以取得 A 股及指数报价；A 股权威日线保存在 PostgreSQL，并可通过 `CNMarketHistoryQueryService` 按复权模式查询。前端已有基于 `klinecharts` 的 K 线组件、技术指标工具和回测中心，但没有 A 股浏览入口。

本变更跨越后端行情聚合、缓存、历史查询和 Vue 页面，且全市场近实时快照不能按股票逐项请求，否则会放大上游流量并拖慢低资源服务器。页面还必须区分近实时报价、已收盘日线以及展示级降级日线，避免用户误判数据时效或回测资格。

## Goals / Non-Goals

**Goals:**

- 为已登录用户提供从市场概览、沪深股票目录到个股详情的连续研究路径。
- 复用现有标的主数据、自选股、本地 A 股日线、报价与技术指标基础能力。
- 使用共享快照支撑列表及市场宽度指标，避免 N+1 上游报价请求。
- 所有行情和指标都携带时间、来源、新鲜度及可用状态。
- 在本地日线不足时保持页面可用，同时明确降级日线不能作为权威回测数据。

**Non-Goals:**

- 北交所、分钟 K 线持久化、全市场日线批量回填。
- PE、PB、ROE、财报、行业排行、资金流向或北向资金。
- 自动启动回测、下单或任何交易执行。
- 改变现有 A 股回测对历史覆盖、质量和市场规则的严格要求。

## Decisions

### 1. 在现有 `/api/market` 人类用户接口下增加 A 股聚合资源

新增读取接口：

- `GET /api/market/cn/overview`
- `GET /api/market/cn/stocks`
- `GET /api/market/cn/stocks/<symbol>`
- `GET /api/market/cn/stocks/<symbol>/history`

全部沿用 `@login_required` 和现有 `{code,msg,data}` 响应约定。列表接口接受 `keyword`、`exchange`、`changeState`、`page`、`pageSize` 和受限排序参数。详情身份信息、报价和历史拆开，使报价的短 TTL 不会迫使日线与指标反复计算。

备选方案是在 `/api/global-market` 中扩展；未采用，因为该模块面向跨市场仪表盘，A 股股票目录、用户自选状态和本地历史属于更具体的市场资源。

### 2. 建立单一共享的沪深市场快照服务

在 `app/services/market/` 增加隔离的 A 股快照适配器和聚合服务。适配器一次取得沪深 A 股近实时快照并标准化为统一字段；概览与股票列表从同一快照派生。Redis 保存短 TTL 新鲜缓存和较长 TTL 陈旧缓存，并通过现有 `guarded_cached`/锁机制合并并发刷新。

刷新 Future 超过配置时限后不得被后续请求永久复用。服务解除对该 Future 的引用，使后续请求可以重新尝试刷新；仍在执行且无法由 Python 强制终止的线程只负责完成原调用，其结果不再阻塞新的请求。这样既保持正常并发下的单次刷新合并，也避免一次长期阻塞令整个进程永久失去刷新能力。

上游实现优先复用当前依赖能够提供的全市场快照接口；核心指数通过批量或受控请求取得。路由和业务层只依赖标准化快照协议，不暴露上游原始字段，便于以后替换数据源。

备选方案是对当前页逐只调用 `get_single_price`；未采用，因为筛选涨跌状态和计算市场宽度需要完整一致快照，逐只请求还会形成 N+1 流量。

### 3. 标的主数据决定目录身份，行情快照只负责动态字段

目录基表使用 `qd_market_symbols` 中启用的 `CNStock` 行，并通过现有规范化规则推导 `SH`/`SZ`，不信任同步数据中笼统的 `exchange='CN'`。行情快照按规范代码连接，缺少行情时仍保留主数据行并返回 `quoteStatus='unavailable'`。搜索、静态交易所筛选在主数据上完成；涨跌状态筛选在同一快照连接后完成；最后执行稳定排序和服务端分页。

当前用户的自选集合只与分页后的结果连接，避免全表用户态查询，并防止跨用户泄露。

部署网络无法取得完整快照时，允许使用腾讯批量接口为当前页提供展示级报价，但该结果不再宣称是完整市场快照。列表响应必须返回 `partial=true`、`quotedCount`、`catalogCount`，并使用 `freshness='partial'` 和降级状态。若用户同时使用涨跌筛选，由于展示级报价只覆盖受限集合，响应还必须返回 `changeStateComplete=false`；前端必须展示“逐页降级行情”及原因，不得把此时的筛选总数表达为全市场统计。

### 4. 市场宽度统计复用现有 A 股价格限制规则

上涨、下跌和平盘按最新价相对昨收判断；总成交额只聚合有效非负成交额。涨跌停判断复用 `cn-backtest-market-rules` 已实现的日期、板块和 ST 分类规则，而不是简单统一使用 10%。无法确认分类或昨收的证券进入 `unclassifiedLimitCount`，不计入涨跌停数。

备选方案是统一按 ±10% 判断；未采用，因为科创板、创业板和 ST 股票会产生明显错误。

### 5. 日线采用“本地权威优先、展示级降级”双层读取

`/history` 默认请求最近约 260 个交易日并支持受限 `limit` 与复权模式。服务先使用 `CNMarketHistoryQueryService` 和质量覆盖结果读取本地历史；覆盖合格时返回 `backtestEligible=true`、`dataVersion` 和因子版本。本地没有覆盖时，调用现有 CNStock 日线展示链路取得日线，并明确返回 `tier='display_fallback'`、供应商和 `backtestEligible=false`。

这不会自动触发管理员同步，也不会放宽回测中心的覆盖门禁。用户从详情进入回测后，如本地权威历史不足，仍会收到现有覆盖错误和定向同步建议。

备选方案包括打开详情即写库或首期全市场批量回填；均未采用，因为普通读取请求不应隐式执行运维写操作，而全市场回填超出首期资源和验收范围。

### 6. 技术指标由后端基于同一日线统一生成，图表复用前端能力

后端扩展现有 `calculate_indicators`，补齐 KDJ，并让每个指标返回参数、最新值、状态、所需数据量和实际数据量。指标缓存键包含规范标识、复权模式和日线数据/缓存版本。`/history` 同时返回 bars、indicators 和 provenance，前端只负责绘图与展示，不再另取一套数据计算摘要。

前端复用 `klinecharts` 和现有技术指标绘制工具，但为详情页提供轻量封装，使图表使用 `/history` 返回的 bars。这样 K 线与指标卡共享同一数据版本。

### 7. 前端新增两个路由并使用显式刷新策略

- `/cn-stocks`：市场概览卡片、筛选栏、分页表格和自选操作。
- `/cn-stocks/:symbol`：报价头部、日 K 线、技术指标卡、来源信息、自选和回测入口。

菜单使用现有 `dashboard` 权限，详情路由隐藏。市场页保持短时页面缓存并在激活或定时器触发时刷新；仅页面可见时轮询，离开页面立即停止。默认刷新间隔不得短于服务端新鲜 TTL，手动刷新也受服务端合并保护。

### 8. 使用结构化部分成功而非全页失败

概览、列表和详情响应分别携带组件级 `status`、`asOf`、`source`、`freshness` 和可选 `warning`。指数、全市场快照或个股历史中的单个来源失败时，页面保留其他有效区域。只有身份参数非法、未认证或核心资源完全不可用时返回请求级错误。

个股详情在完整快照不可用时可以使用腾讯单标的批量报价降级；若该降级源也失败，服务必须统一转换为 `CNMarketSnapshotUnavailable` 并由路由返回结构化 `503`，不得将上游网络异常泄漏为未处理的 Flask `500`。

## Risks / Trade-offs

- [全市场快照源在境外服务器上可能不稳定] → 使用短时共享缓存、陈旧数据窗口、超时和结构化降级；验收时覆盖上游失败场景。
- [快照股票与主数据代码格式不一致] → 两侧先规范化为 `CNStock:<code>.<exchange>` 再连接，并报告未匹配数量。
- [全市场涨跌停分类数据不完整] → 复用已确认规则；无法确认的证券不计入并公开未分类数量。
- [本地日线当前只覆盖少量标的] → 允许明确标记的展示级降级，但绝不赋予回测资格或隐式写库。
- [现有 K 线组件功能较重] → 通过轻量封装复用绘图核心，避免把指标 IDE 的编辑、Pyodide 等全部耦合到详情页。
- [低内存服务器构建前端可能耗时] → 后端与前端按项目约束分别单任务构建、测试和部署，绝不并发构建。

## Migration Plan

1. 先增加后端快照、目录、详情和历史聚合服务及接口测试，不修改现有接口语义。
2. 增加前端 API 客户端、路由、页面、国际化文案和组件测试。
3. 运行小范围后端测试和前端静态/目标测试；随后按低内存规则先构建部署后端，再构建部署前端。
4. 验收登录权限、正常行情、陈旧缓存、单项缺失、搜索筛选分页、详情日线来源、自选同步和回测预填。
5. 回滚时先回滚前端入口，再回滚新增后端路由和服务；本变更不做破坏性数据库迁移，现有回测和运维功能不受影响。

## Open Questions

无阻断问题。具体全市场快照上游在实施时通过适配器验证；若当前依赖的主源在部署网络不可达，则保持相同协议切换现有可用源，不改变产品规格。
