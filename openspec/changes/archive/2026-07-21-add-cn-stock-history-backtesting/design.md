## 背景

当前 `CNStockDataSource` 通过 Twelve Data、腾讯、yfinance 和 AkShare 的降级链即时获取 K 线。Strategy API V2 的 `market_data.py` 根据时间跨度估算 `limit`，将结果放入进程内 `KlineCache`，但无法证明返回数据覆盖了请求区间，也无法在多进程、重启或供应商数据变化后复现同一次回测。通用撮合器目前主要使用统一比例佣金和滑点，尚未表达 A 股 T+1、交易单位、涨跌停和税费差异。

本变更横跨 PostgreSQL 数据模型、外部行情采集、后台任务、Strategy API V2 回测、管理员 API 和独立 Web 前端。运行主机资源有限，所有采集、迁移、测试和镜像构建必须遵循单任务、低内存、磁盘受控的约束。`docs/agent/*` 仍保持英文，本变更的 OpenSpec 文档使用中文。

## 目标 / 非目标

**目标：**

- 为 A 股普通股票提供可增量维护、可审计、可复现的本地原始日线库。
- 对用户区间和预热区间进行失败关闭式覆盖校验，使回测不再静默接受截断数据。
- 在不破坏其他市场通用运行时的前提下，通过市场执行策略表达基础 A 股交易规则和费用。
- 为管理员提供可观察、可恢复、可定向控制的同步流程，并在回测结果中暴露数据与执行假设。
- 使用现有 PostgreSQL、Celery 和 Flask 边界，保持部署拓扑简单并符合低资源主机约束。

**非目标：**

- 不建设全市场分钟级或逐笔数据库，不引入新的数据库服务。
- 不承诺覆盖 ETF、可转债、B 股、北交所特殊品种或所有历史规则变迁；首期只支持已明确分类的沪深普通 A 股。
- 不用本地历史库替换行情页的实时快照和盘中 K 线降级链。
- 不提供 A 股实盘下单，也不放宽现有交易安全边界。
- 不在本主机上构建 Mobile/H5，不执行全市场一次性高并发回填。

## 决策

### 1. 在现有 PostgreSQL 中保存规范化原始数据

新增以规范标的、交易日和数据源为核心键的日线表，并分别保存公司行动、复权因子、同步批次、单标的检查点、质量问题和覆盖摘要。原始价格不覆写，复权价格在查询阶段由因子确定性生成。

选择 PostgreSQL 是因为项目已有连接、迁移、备份和权限体系，日线规模在首期可控。相比之下，引入 ClickHouse/MongoDB 会增加部署与运维成本；按文件保存 Parquet 虽适合分析，但不利于事务式增量更新、管理员查询和多实例一致性。

### 2. 使用适配器隔离 `easy_tdx`，供应商数据只写原始层

新增 TDX provider 适配器，负责连接生命周期、分页、节流、错误分类和原始字段转换，并复用 easy_tdx 已提供的节点优选、重连、健康评分及空数据故障转移。业务层只依赖内部 DTO，不直接依赖 easy_tdx 类型，也不导入 `easy_tdx._health`、`easy_tdx._reconnect` 等私有模块。

开发阶段暂不向 PyPI、GitHub Release 或其他公共包仓库发布 easy_tdx。QuantDinger 从本机 sibling 仓库 `/home/quantadinger/easy_tdx` 的固定基线构建 wheel 后安装；当前基线为 easy_tdx `1.20.4`、Git 提交 `3b7ce97`。wheel 是开发制品，不将 sibling 源码复制进 QuantDinger，也不以 editable install 或运行时源码挂载作为生产依赖。正式发布和生产制品来源在发布阶段另行决策。

TDX 是首期主要批量来源，因为其日线拉取适合中国市场且可分页；现有腾讯等来源继续服务实时/图表降级，但不得在一次本地历史回测中静默混合补洞。这样可避免不同供应商口径被伪装为单一确定性数据集。

### 3. 以单标的检查点和数据库锁实现可恢复串行同步

每次同步创建批次记录，每个标的独立记录目标区间、游标、状态、重试次数和错误。任务按标的串行执行，小批量提交；同一同步域使用 PostgreSQL advisory lock 防止多 worker 重叠。失败后从最后已提交检查点继续，不回滚其他已成功标的。

选择数据库锁而不是仅依赖 Celery 队列唯一性，是因为多个进程或重复投递仍可能并发执行；选择逐标的事务而不是全批次事务，可限制锁时长、内存占用和失败回滚成本。

### 4. 覆盖率由交易日历和质量状态共同判定

覆盖校验基于 `market_schedule.py` 所使用的 XSHG 交易日历生成预期交易日，并区分上市前、退市后、停牌、尚未收盘和真正缺失。仅有最早/最晚日期不足以证明连续覆盖，因此覆盖摘要用于快速筛选，最终执行前仍查询缺口和阻断级质量问题。

若无法证明请求区间（含预热）完整，回测必须返回稳定的错误码、缺失区间、受影响标的及建议同步动作。系统不在回测请求内自动发起远程补数，以保持延迟、权限和可复现性可控。

### 5. 交易日使用上海时区的无时区日期语义

日线表以 SQL `DATE` 保存交易日。加载成 pandas 索引时，将交易日映射为不带时区的本地会话标签，不再先按 UTC 午夜解释再换算导致日期漂移。回测输入边界仍接受现有 API 日期/时间格式，但 A 股日线选择按 `Asia/Shanghai` 交易日截取。

### 6. 用市场执行策略扩展通用撮合器

在 Strategy API V2 通用运行时中引入 `MarketExecutionPolicy` 边界。默认策略保持其他市场现有行为，`CNStock` 策略负责：仅做多、100 股买入单位、清仓零股、T+1 可卖数量、无有效价格/停牌拒单、按证券分类和日期确定涨跌停、以及分项费用计算。

相比在通用撮合循环中散布 `if market == "CNStock"`，策略对象更便于单元测试和后续扩展。相比建立独立 A 股回测引擎，复用统一事件调度、策略契约、结果存储和保护逻辑可减少行为分叉。

首期日线回测继续在下一根可交易 bar 的开盘价撮合；若该价格触及禁止方向的涨跌停或 bar 无成交能力，则订单延期并受既有最大尝试次数约束。成交费用必须记录佣金、最低佣金补差、印花税、过户费和总额，而不是只返回一个不可解释的比例费用。

### 7. 原始数据、派生因子和回测快照分别版本化

日线和公司行动行记录供应商、采集时间及内容校验信息；复权因子记录算法版本和生成时间；回测结果记录数据集版本/覆盖摘要、复权模式、同步批次或快照标识及市场规则版本。历史修订发生后，新回测使用新版本，旧回测仍能说明当时输入。

完整复制所有日线到每个回测快照会迅速消耗磁盘，因此首期以内容版本和覆盖水位作为默认来源证明，仅在现有内容寻址快照机制明确启用时生成物理快照。

### 8. 管理接口与普通回测接口严格分权

同步启动、重试和供应商诊断仅允许管理员访问，并写入操作者与批次审计信息。普通回测用户只可读取自身回测所需的覆盖错误和公开假设，不可借回测请求触发任意全市场同步。前端设置页只调用管理员端点，回测中心只展示来源、覆盖和规则。

### 9. 磁盘保护优先于同步完整性

同步启动前及每个标的小批次提交前检查数据库/根文件系统可用空间。达到软阈值时不再接受扩大覆盖的新任务；达到硬阈值时安全暂停当前批次并保留检查点。阈值通过环境变量配置并在管理员状态接口显示。

不自动删除旧行情、Docker 缓存或共享缓存，因为这些操作可能破坏可复现性或影响其他服务。释放空间必须由管理员明确执行。

### 10. 使用交易所结构化参考价验证除权参数

TDX 继续提供原始公司行动，但复权因子发布前必须使用官方结构化数据验证除权日参考价或除权参数。沪市读取上交所公司分红接口的 `PRE_CLOSE_PRICE`；深市优先读取深交所除权日行情的 `qss`，当该历史行情归档尚未覆盖目标日期时，读取巨潮资讯结构化历史分红接口的除权日、派息和送转参数。保存来源 URL、响应内容哈希、验证时间、官方参考价或参数及最终事件比例。

普通事件先按 TDX 字段计算精确理论参考价，并按 A 股价格精度与官方参考价比较；一致时保留 TDX 精确比例。深市行情归档滞后时，只有纯现金事件且 TDX 参数与巨潮官方每股派息在容差内一致，才使用官方精确派息参数计算比例。若纯现金分红不一致，允许按官方参考价与除权前原始收盘价推导经审计的事件比例，以覆盖差异化分红。包含送转、配股、缩股等复合事件且参考价或参数不一致时失败关闭，不从两位小数参考价反推多个未知参数。

选择结构化交易所参考价而不是把 PDF 文本解析作为主路径，是因为前者字段稳定、体积小、便于哈希审计和自动重试。公告 PDF 仍是人工对账证据，但不作为日常同步成功的唯一条件。

## 风险 / 权衡

- [TDX 公共节点不稳定或限流] → 多节点健康评分、有限重试与退避、请求节流、检查点续传，并在管理界面显示最近成功时间。
- [供应商历史数据被修订] → 保留采集元数据和版本标识，重算受影响因子；回测结果记录数据版本，不覆盖既有结果说明。
- [交易日历无法区分停牌和缺数] → 结合证券状态/零成交记录和质量规则；不能证明为合法缺失时按覆盖缺口失败关闭。
- [公司行动口径不完整造成复权错误] → 原始价永不覆写，因子带算法版本，异常跳变进入阻断级质量问题，管理员可重新同步和重算。
- [差异化分红的实际派发额与交易所除权参数不同] → 2026-07-21 权威样本对账确认 `easy_tdx` 只提供贵州茅台实际每股派发额 27.673 元，而上交所除权口径使用摊薄后的 27.600 元；现已接入交易所结构化官方参考价，纯现金差异化事件使用审计比例，复杂不一致事件仍阻断。
- [通用撮合器扩展引发其他市场回归] → 默认策略保持原行为，A 股规则采用独立策略并增加市场隔离回归测试。
- [全市场数据耗尽磁盘] → 首期仅同步明确标的/系统标的池，串行小批量写入，设置软硬磁盘阈值且不支持分钟线导入。
- [最低佣金导致按比例现金上限估算不准确] → A 股策略用分段费用函数反算最大可买数量，并在撮合前后进行现金校验。
- [前后端分属不同仓库导致发布不一致] → API 返回显式能力/版本字段，前端对未提供新端点的旧后端做降级展示，分别提交和发布。

## 迁移计划

1. 从 `/home/quantadinger/easy_tdx` 的固定开发基线构建本地 wheel，验证版本、提交、校验值、许可证和 Python/pandas 兼容性；在后端依赖层加入适配器但默认不开启同步。
2. 同时更新日期迁移和 `migrations/init.sql`，先部署可回滚的新表、索引和配置；不修改旧表中的行情数据。
3. 部署 provider、仓储、质量与同步服务，以少量测试标的执行串行回填和数据对账。
4. 部署管理员只读状态 API，再开放定向同步；观察节点健康、失败率、数据库和磁盘增长。
5. 为 Strategy API V2 增加本地 A 股加载和覆盖校验，通过功能开关先在测试环境启用；其他市场继续使用原加载路径。
6. 启用 A 股执行策略和分项费用，验证基准案例后在回测响应中发布规则版本。
7. 部署 Web 设置页与回测中心展示，最后按选定系统标的池逐步扩展覆盖。

回滚时关闭本地历史/同步功能开关并停止相关 Celery 调度，Strategy API V2 恢复旧数据加载路径；新增表保留以便审计和再次上线。若迁移本身失败，只回滚本次新增对象，不删除已成功采集的数据，除非管理员明确批准。

## 待确认问题

- 首批“系统标的池”的明确来源和规模，需要在实施同步调度前由产品/运维确认。
- easy_tdx 公司行动接口对长期历史的完整性，需要用权威样本对账；若不足，后续可增加第二供应商，但不得静默混源。
- 不同券商的佣金率可配置，首期默认值及过户费历史生效日期需要在上线前以当前业务口径确认。
- 科创板、创业板、ST 与退市整理期的证券分类历史是否可从现有标的主数据可靠获得；无法可靠分类的区间应阻断回测，而不是套用普通 10% 限价。
